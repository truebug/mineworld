## 1. 我们要解决什么问题

### 1.1 两侧各自的痛点

| 侧 | 强项 | 痛点 |
|----|------|------|
| **Godot** | 关卡/任务/地图编辑、UI、相机、输入、原生 3D、导出多平台 | 机甲个体物理偏「游戏级」，难以支撑关节级真动力学与科研级数据采集 |
| **MuJoCo** | 无头高精度物理、关节/接触、适合机器人仿真 | 世界模型创建与编辑成本高，缺少面向玩家的任务/叙事/交互壳 |

### 1.2 核心诉求（归纳）

1. **用无头 MuJoCo 支撑机甲物理仿真**（关节动作等），替代游戏引擎内对机甲的简陋动作/物理计算。
2. **用 Godot 承担世界编辑与可视化 Viewer**，降低 MuJoCo 侧世界搭建复杂度。
3. **玩家可接管 MuJoCo 侧机甲的人工遥操**，遥操数据被采集记录。
4. **成品可在娱乐（玩游戏）环境下**完成：遥操数据采集、与可变交互世界交互结果的数据采集。
5. 在此基础上延伸多种**学习模式、娱乐模式、商业模式**。
6. **作为数聚球仿真生态的 3D 传送门前台**：本仓关卡跑真动力学；展厅/教室/实验室等经 Hub 通道进入 PMS 既有 Space/卡片（见 [21](21-ecosystem-federation.md)）。

## 2. 产品定位

### 2.1 不是什么

- 不是复刻《头号玩家》全感官虚拟人生或大规模元宇宙社交。
- 不是用游戏引擎内置物理硬扛机器人动力学。
- 不是让策划去手写 MJCF 搭完整开放世界。
- **不是**把 PMS / Gazebo / 高保真场景编排搬进本仓 Gateway（对接，不搬迁）。

### 2.2 是什么

**太空机器人学院 + 竞技场** 的初始底座（开源叙事：**机甲学院母港**），并可作为数聚球 **统一前台里的传送门主体**：

- 可编辑的共享仿真游乐场（Godot Hub + 本仓关卡）
- 真物理机体（MuJoCo 权威 · 本仓玩法）
- 可回放的人类行为档案（录制管道）
- **外部卡片通道**：展柜/课件 → PMS Space（mjviser / GZWeb / WebIDE 等既有能力）

对外叙事：**真机体、真任务、真遥操数据** —— 学员在母港训练与竞技；产品侧采集：

| 数据意图 | 说明 |
|----------|------|
| **本体 / 关节轨迹** | `cmd` ↔ `joints` / 接触事件，服务模仿学习与「小脑」级控制 |
| **任务约束示范** | 成功/失败可筛，按关卡与模式打标签 |
| **仿真视觉（后置）** | 相机流与同步状态入库；本期不抢跑 |

**远景任务谱**（搬箱、积木、找路、竞速、越障、探图、格斗、船/机、球类、生存等；solo / FFA / 组队 / 决斗 / 计时）是北极星，**不等于本期全部开做**。近端仍服从 [16](16-value-sprint.md)：工坊臂爪 IL 做深，再横向扩关卡与赛事壳（排期见 [09](09-todo.md)）。

对外一句话：**真机体、真任务、真遥操数据** + **可发现的仿真宇宙入口**；世界房间与展柜慢慢挂，不抢跑。

### 2.3 价值飞轮

```
好玩 → 更多人玩 → 更多有任务约束的遥操数据
  → 更好的策略/演示 → 更好玩 / 更好卖数据或算力
生态发现（Hub/Spaces）→ 更多进入 PMS Space / 本仓关卡 → 同一身份下的会话与资产
```

娱乐与 3D 导览是获客入口；本仓仿真与数据是核心资产；PMS 是执行农场。

## 3. 角色与使用场景

| 角色 | 使用 Godot | 使用 MuJoCo | 产出 |
|------|---------------|-------------|------|
| 关卡/策划 | 编辑场景、任务、触发器、UI | 不直接编辑 MJCF（通过场景契约） | 可玩关卡 + 场景契约 |
| 仿真工程师 | 验收映射与回放 | MJCF、控制栈、网关 | 稳定仿真与协议 |
| 玩家 | 进关、操控、看反馈 | 间接（权威状态来自 MuJoCo） | 遥操轨迹、任务结果 |
| 算法/数据 | 读录制包 | 离线重放/评测 | 训练集、评测榜 |
