### 定位校准 · 2026-07-19（数据价值）

**最初设想**：娱乐壳获客 → MuJoCo **真机体** 权威 → 有任务约束的**人类遥操档案**（关节/接触可训）。

**当前 POC 实际**：平面 DiffBot + `velocity(vx,vy,ω)` 轻操控，更像遥控小车/履带车在街区开到绿点。管道（契约、录制、回放、导出）已通，但**控制语义过浅**，单条轨迹对 IL/机器人策略的信息密度低——这不是自动驾驶产品，却也还没回到「机甲关节遥操」叙事。

| 提高数据价值（不偏离愿景） | 说明 |
|---------------------------|------|
| **换控制模式** | `joint_targets` / 轮臂混合；录 `joints`+`cmd` 成对（T2.6 已铺路） |
| **换机体 DoF** | 多关节 mech（非 planar slide）；Godot 跟皮已有 F3/F8 |
| **加接触任务** | 推箱升级：堆叠、门、抓取点；失败/纠偏会话 |
| **任务标签** | outcome + 子目标事件写入 header，按难度分层 |
| **避免的偏航** | 不要做成车道保持/规划 benchmark；价值在**人控真动力学**，不在 AD 感知 |

铁律不变：Godot 不权威、MuJoCo 不叙事。下一阶段优先 **控制深度**，不是再扩城市皮。

## 1. 一句话结论

**主线正确、节奏健康、集成风险已被压住；下一步应收束到「可玩闭环（M4）」而非继续扩关卡/扩能力。**

双引擎 + Gateway 权威 + 本地录制这一拓扑经得起 POC 验证；最大缺口已从「能不能通」变为「有没有一关能玩完并留下可信数据」。

| 维度 | 评级 | 一句话 |
|------|------|--------|
| 产品定位 | **强** | 「娱乐壳采真物理遥操数据」叙事清晰，Out of Scope 克制 |
| 技术路线 | **强** | 契约/协议/Schema 先行，假物理→真物理换后端不改客户端 |
| 开发计划 | **良** | 里程碑清楚；Phase 2 末段与 Phase 3 前半有交叉，需再排优先级 |
| 工程产出 | **良** | M1–M3 实质达成；单文件 Gateway 已开始膨胀，需防「第二叉路」 |
| 数据资产就绪度 | **中** | 能录能画轨迹；缺任务标签、脱敏、批量评测接口 |

## 2. 原始设计（冻结意图）

### 2.1 要解决的问题

| 侧 | 痛点 | 设计回应 |
|----|------|----------|
| 游戏引擎 | 机甲物理偏「游戏级」，难支撑关节级动力学与科研级数据 | 权威交给无头 MuJoCo |
| MuJoCo | 世界编辑成本高，缺任务/叙事壳 | Godot 做关卡与 Viewer |
| 产品 | 好玩与可信数据往往分家 | 在同一会话里：遥操 → 真物理 → 落盘 |

### 2.2 架构铁律（至今未破）

1. **Godot 不权威**：只发 `cmd`、展示 `state`，不写录制。
2. **MuJoCo 不叙事**：不做关卡编辑器、不做账号。
3. **Gateway 是唯一边界**：会话、契约、控制映射、广播、录制旁路。
4. **禁止叉路**：不在 Godot 内嵌 MuJoCo；不用引擎 Multiplayer 同步位姿；不在客户端落盘。

### 2.3 POC 冻结默认（D1–D7，已 Accepted）

米 · 右手 · **Z-up** · `velocity` · Python Gateway · `dt=0.02` / sim 50Hz / state 20Hz · 本地 `recordings/` · **自建盒子机甲** · 手写契约。

这些默认值把「选型争论」挡在实现门外，是本项目推进快的核心原因。

## 3. 迭代过程（事实时间线）

约 **3 个日历日**（2026-07-17 → 07-19），从空目录到真物理 + 可录：

```text
Day 0–1  愿景/架构文档 → Schema SSOT → ADR 冻结
         → Gateway echo + 假积分 → GDevelop demo0 打通 M1
Day 1    Godot spike 复验 M1 → ADR-003 换引擎（沉没成本最低窗口）
Day 1–2  tutorial_02 场景 + Kenney 资产 + 契约（关卡工作流验证提前做）
Day 2–3  T2.1 MJCF → T2.2 MujocoMech → T2.3 契约障碍 → T2.5 录制
```

### 3.1 关键转折

| 转折 | 决策 | 评价 |
|------|------|------|
| **假物理同形协议** | POC-A 用 kinematic 积分，消息形状与真仿真一致 | 优秀：换 MuJoCo 时客户端几乎零改 |
| **GDevelop → Godot** | ADR-003，在 POC-B 前切换 | 正确且及时：3D/插值/无头 CI 是硬需求；UGC 无代码优势 MVP 用不上 |
| **tutorial_02 前置** | 真仿真未齐时先做城市关 + 资产台账 | **有收益也有代价**：验证了关卡工作流与授权；但与「最短路径先闭 M3」并行，注意力被分走 |
| **fake 后端保留** | `--physics fake\|mujoco` | 正确：回归与无 MuJoCo 环境仍可冒烟 |

### 3.2 未按原「5 工作日 POC」线性走完的部分

原计划 POC-B ≈3 天含 take_control UI + 录制。实际：

- 真物理 + 障碍 + 录制已齐（M2 + 最小 M3 ✅）
- `take_control` **协议层已有**，客户端 join 后自动接管；缺的是释放 UI / 事件体验打磨（T2.4）
- **任务闭环**（终点判定 → `objective_complete`）仍空（T3.1 → M4）

结论：工期叙事仍成立，但「可玩」比「可录」晚半步——这是当前最大产品缺口。

## 5. 方案审核

### 5.1 产品设计

**做得好的**

- 价值飞轮写清楚：好玩 → 数据 → 更好的演示/策略 → 更好玩。
- 对外叙事收窄为「真机体、真任务、真遥操数据」，避免元宇宙膨胀。
- MVP 明确「一关、一机甲、接管、落盘」，与当前工程规模匹配。

**需要警惕的**

1. **「关卡好看」抢戏「任务可完成」**  
   tutorial_02 已证明拼关能力，但没有 `objective_complete` 时，数据缺少成功/失败标签，对 IL / 评测价值打折。  
   **建议**：下一阶段禁止新开第三关，直到 M4 在 tutorial_01（或 02 二选一）闭合。

2. **盒子机甲 vs 真实机器人期望**  
   产品故事容易滑向「人形/四足真机」，POC 盒子是对的，但对外演示要主动降预期，避免干系人以为「下周接 g1」。  
   **建议**：在 README / 演示脚本固定一句：「POC 权威链路用盒子；真实 MJCF 是 P1 换皮，不改协议。」

3. **人/机同视图（`04` §5.1）写得早、落得晚**  
   方向正确（T2.6 / T4.5），但不应挡在 M4 前。传感器出口可以在「能玩完一关」之后再补。

### 5.2 技术路线

**做得好的**

- Schema 可扩展信封（禁止 v0 `additionalProperties: false`）降低协议改崩风险。
- 坐标映射集中在傀儡一处（`godot_pos = (x, z, -y)`），符合「边界清晰」。
- 录制落在 Gateway，语义与 WS state 同构，回放脚本已能消费。

**技术债与风险（按优先级）**

| # | 风险 | 现状 | 建议 |
|---|------|------|------|
| R1 | **单进程单 `MjData`** | 多会话会互相踩 | M4 前可忍；进多客户端前必须一会话一 `MjData` 或进程隔离（T4.1） |
| R2 | **`echo_server.py` 过大** | 物理 + WS + 契约编译挤一文 | M4 后拆 `physics/`、`session/`、`ws/`；现在只抽 recorder 已开了个头 |
| R3 | **契约与 `.tscn` 双源** | 手写 JSON 易漂移 | 短期纪律：改关必改契约；中期 T4.2 插件导出 |
| R4 | **录制体积** | 每 tick 全量 state，10s ≈ 数百 KB–MB | POC 可接受；上规模再 `record_every_n_ticks` / 列存 |
| R5 | **无自动化 CI** | 冒烟靠本机脚本 | 加最小 GitHub Action：`fake` 物理下 `ws_smoke_test`（不强制 MuJoCo runner） |
| R6 | **Godot 版本漂移** | 本机可能升到 4.7 改 `project.godot` | 锁定文档声明版本；勿把无关 `.import` 噪声提交进主线 |

### 5.3 开发计划与优先级

原最短路径：

```text
决策 → Schema → Gateway → 客户端 → MuJoCo → 录制 → 任务闭环
```

执行上大体遵守，但插入了「换引擎」与「第二关资产」。换引擎值得；第二关应视为**工作流 spike**，不要当成 Phase 3 已完成。

**建议的下一窗（2–4 个工作日，收束 M4）**

```text
1. T3.1  Gateway：AABB/触发器 → objective_complete 事件（权威在服务端）
2. 客户端  最小结算 HUD（成功/失败文案即可）
3. T2.4  补 release_control + 简单 UI（可与 2 同批，勿单独拖期）
4. 录制   确认 outcome=success|fail 写入 header（T2.5 已具备钩子）
5. 回归   fake + mujoco 双冒烟；≥10s 成功局可 replay
```

**明确后置（不要穿插）**

- T2.6 传感器字段、T2.7 延迟补偿 —— 体验/AI 准备，非 M4 阻塞  
- T3.3 新关、T4.\* 扩展 —— M4 演示视频拍完再动  
- 账号 / TLS / 对象存储 / 合规模板 —— 商业前再开专项

### 5.4 整体方案是否成立？

**成立。** 证据：

1. 同一客户端协议下，`--physics fake|mujoco` 可切换（权威边界真实存在）。
2. 契约障碍进入 MuJoCo 世界后有碰撞反应（双世界映射不是纸面）。
3. 会话可落盘且轨迹可回放（数据采集管道不是口号）。
4. 引擎可替换（GDevelop→Godot）而不改 Schema/Gateway 核心 —— 说明集成边界选对了。

尚未证明、但不必在 POC 证明的：多副本扩展、真实人形动力学、UGC 编辑器、商业合规。保持 Out of Scope 是正确的产品纪律。

## 9. 结语

MineWorld 没有做成「又一个机甲游戏原型」，而是做成了一条**可替换物理后端的遥操数据管道**——这在 POC 阶段是更高价值的形态。  

当前最该避免的失败模式不是技术选错，而是：**在任务闭环完成前继续堆关卡、传感器与扩展特性**，让「能录」停留在「无标签轨迹」。  

收束 M4，再谈放大。
